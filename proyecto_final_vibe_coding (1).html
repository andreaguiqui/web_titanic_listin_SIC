Chapter 8 · Proyecto final · Guion + enunciado

Streamlit Data Studio: dos datasets, nueve formas de visualizar

Construir con Vibe Coding una app multipágina que responda preguntas reales sobre el Titanic y el alojamiento turístico de Madrid, y demostrar criterio para elegir cada visualización.

Formato parejas Tiempo 2 sesiones de 3 h Entrega app + repo + memoria IA Defensa 5 min

00
La misión en una frase

Crear un laboratorio visual reproducible: la misma app debe permitir explorar dos historias distintas, justificar por qué cada gráfico es adecuado y registrar cómo se utilizó la IA para llegar a un resultado que el equipo entiende y puede defender.

Portada · decisiones

Data Studio

KPIs, selector de dataset, calidad del dato y conclusiones principales.

Historia 01 · Titanic

¿Quién sobrevivió?

Clase, sexo, edad, tarifa y composición familiar.

Historia 02 · Madrid

¿Dónde y a qué precio?

Distribución espacial, barrios, tipo de habitación y disponibilidad.

Resultado esperado

No se evalúa cuántos gráficos caben en una pantalla. Se evalúa si la app permite comparar, filtrar, descubrir y explicar, y si el equipo puede señalar qué código propuso la IA, qué corrigió y por qué.

01
Los dos datasets

Obligatorio · dataset 1

Titanic

891 pasajeros, variables categóricas y numéricas, valores ausentes y una pregunta objetivo clara. Usar el CSV de ejemplo de Seaborn: survived, pclass, sex, age, fare, embarked, class y alone.

Recomendado · dataset 2

Alojamientos de Madrid

El resumen listings.csv de Inside Airbnb añade la dimensión geográfica que Titanic no tiene: latitude, longitude, neighbourhood, room_type, price, minimum_nights y availability_365.

Fuentes: Titanic · seaborn-data y Madrid · Inside Airbnb. Descargar una copia y guardarla en data/; la app no debe depender de internet durante la demo.

Por qué dos

Titanic funciona muy bien para comparación, distribución y relaciones; Madrid permite mapas, capas y densidad espacial. Forzar un mapa sobre Titanic obligaría a inventar coordenadas y degradaría el proyecto.

Criterio de diseño

“Usar todas” significa demostrar su punto fuerte. La galería puede contener ejemplos compactos; la narrativa principal debe conservar solo los gráficos que ayudan a responder una pregunta.

07
Prompt 1 · construir el esqueleto

Actúa como desarrollador senior de Streamlit. Crea el esqueleto de una app
multipágina para explorar Titanic y alojamientos de Madrid.

Restricciones:
- Python, Streamlit y pandas; no añadas todavía gráficos.
- Carga CSV desde data/ con funciones decoradas con @st.cache_data.
- Valida columnas obligatorias y muestra un st.error comprensible si faltan.
- Sidebar con navegación, filtros globales y botón para restablecerlos.
- No inventes columnas. Usa exactamente los esquemas que incluyo debajo.
- Devuelve primero el árbol de archivos y después el contenido de cada archivo.

Criterios de aceptación:
- `streamlit run app.py` arranca sin warnings propios.
- Una página no vuelve a leer el CSV en cada interacción.
- Los filtros vacíos producen una vista válida, no una excepción.

08
Prompt 2 · historia Titanic

Añade la página Titanic sin modificar la carga de datos.

Preguntas: supervivencia por sexo y clase; distribución de edad; relación
edad-tarifa; valores ausentes. Implementa:
1. KPIs con st.metric: pasajeros, supervivencia, edad mediana y tarifa mediana.
2. st.bar_chart para tasa de supervivencia agrupada.
3. Seaborn + st.pyplot para heatmap de correlaciones.
4. Altair para edad con color=survived, tooltip y selección.
5. Vega-Lite para composición por clase.
6. Plotly para una vista jerárquica o facetada.

Filtros: sexo, clase, rango de edad y puerto de embarque.
Todos los gráficos deben usar el mismo DataFrame filtrado.
Antes del código, explica qué pregunta responde cada gráfico.

Carga cacheada

@st.cache_data
def load_titanic(path):
    df = pd.read_csv(path)
    required = {"survived", "sex",
                "pclass", "age"}
    missing = required - set(df.columns)
    if missing:
        raise ValueError(f"Faltan: {missing}")
    return df

Métrica correcta

rate = filtered["survived"].mean()
st.metric(
    "Supervivencia",
    f"{rate:.1%}",
    help="Sobre el filtro activo"

Prompt 3 · historia geográfica de Madrid

Añade la página Madrid usando latitude, longitude, neighbourhood, room_type,
price, minimum_nights y availability_365.

Implementa filtros por barrio, tipo de habitación, precio y disponibilidad.
Todos deben afectar a:
- st.map como vista rápida;
- PyDeck con una capa HexagonLayer o ColumnLayer;
- Folium con MarkerCluster y popup de barrio, tipo y precio;
- Plotly scatter precio-disponibilidad;
- st.area_chart o st.line_chart sobre una agregación ordenada.

Limita o muestrea los marcadores si el filtro devuelve más de 3.000 filas.
No expongas datos personales. Mantén el mapa centrado en Madrid.
Comparación que deben explicar

st.map resuelve la vista rápida; PyDeck muestra densidad y capas; Folium aporta marcadores, agrupación y popups. Tres mapas, tres propósitos diferentes.

Prompt 4 · galería y arquitectura

La tercera página documenta el criterio técnico. Debe poder compararse el mismo dato con más de una herramienta sin interferir con las dos historias principales.

Crea una página "Galería" con pestañas: Nativos, Matplotlib/Seaborn, Altair,
Vega-Lite, Plotly, Mapas y Arquitectura.

En cada pestaña incluye:
- un ejemplo ejecutable con datos ya cargados;
- una frase "úsalo cuando" y una limitación real;
- el tiempo de render medido con time.perf_counter;
- un expander con el fragmento de código relevante.

En Arquitectura usa st.graphviz_chart para representar:
CSV -> validación -> cache -> filtros -> agregación -> visualización.
No dupliques la carga ni copies DataFrames completos sin necesidad.

Calidad visual y accesibilidad

  • Título que expresa la pregunta, no el tipo de gráfico
  • Unidades visibles: euros, años, porcentaje
  • Misma codificación de supervivencia en toda la app
  • Paleta legible sin depender solo de rojo/verde
  • Tooltips con contexto, no todas las columnas
  • Orden lógico de categorías y barras
  • KPIs calculados sobre el filtro activo
  • Tablas descargables para verificar resultados
  • Texto alternativo o descripción junto a mapas complejos
  • Layout usable a 390 px y a 1440 px

13
Calidad del dato y ética

Ausentes

No esconderlos

Mostrar cuántos valores faltan, decidir si se excluyen o imputan y explicar el impacto en cada gráfico.

Sesgo

No confundir relación y causa

El Titanic es histórico y descriptivo. Una diferencia entre grupos no demuestra por sí sola una causa.

Fuente

No confundir oferta y mercado

Inside Airbnb compila anuncios públicos; no representa toda la vivienda ni una medición oficial de ocupación.

Prohibido

No construir un “predictor de quién merece sobrevivir” ni presentar datos de alojamientos como prueba de conducta individual. El objetivo es visualización descriptiva y pensamiento crítico.
